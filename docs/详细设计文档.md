本文档用于说明 `tj_server` 多模块 Spring Boot 项目的：

2. 各模块职责边界与依赖关系

## 2. 系统总体设计

### 2.1 项目定位

该项目是一个以“淘客/返利/会员/积分商城/本地生活”为核心的服务端系统，面向以下业务场景：

- APP / H5 / 小程序用户登录与用户中心
- 多电商平台商品检索、商品详情、转链、订单采集
- 会员充值、实名、签约、提现与支付通道管理
- 积分商城、拼团、砍价、秒杀、优惠券等营销玩法
- 后台系统管理、素材管理、配置管理、任务调度与日志监控
- 本地生活 / 外卖 / 抖音本地生活 / 美团 / 饿了么接入

### 2.2 技术栈

| 类别 | 技术选型 |
|---|---|
| 语言 | Java 8 |
| 框架 | Spring Boot 2.6.7 |
| ORM | MyBatis-Plus 3.5.x + PageHelper |
| 缓存 | Redis |
| 数据源 | Druid |
| 文档 | Swagger2 + Knife4j |
| 安全 | Spring Security（后台）/ 自定义拦截器 + JWT（移动端） |
| 定时任务 | Quartz |
| JSON | Fastjson |
| 对象映射 | Dozer |
| 存储 | 本地存储、阿里 OSS、七牛云、图床 |
| 外部平台 | 大淘客、好单库、淘宝联盟、京东联盟、拼多多、抖音、唯品会、美团、饿了么、微信、支付宝、AdaPay、银盛、易宝、通联等 |

### 2.3 运行入口

项目有两个主启动入口：

| 启动类 | 模块 | 作用 |
|---|---|---|
| `mshop-admin/src/main/java/com/mailvor/AdminServer.java` | `mshop-admin` | 后台管理服务入口 |
| `mshop-app/src/main/java/com/mailvor/AppServer.java` | `mshop-app` | 移动端 API 服务入口 |

### 2.4 实际访问基址

根据当前源码中的配置文件：

| 服务 | 配置文件 | 端口 | Context Path | 最终基址 |
|---|---|---:|---|---|
| 后台管理服务 | `mshop-admin/src/main/resources/config/application.yml` | `8001` | `/admin` | `http://host:8001/admin` |
| 移动端 API 服务 | `mshop-app/src/main/resources/config/application.yml` | `8008` | `/api` | `http://host:8008/api` |

> 注意：`RUNNING.md` 中写的是后台 `8000`、应用 `8001`，与当前源码配置不一致。上线前应以实际环境配置为准。

### 2.6 核心调用链

#### 2.6.1 移动端调用链

```text

   -> mshop-app Controller
   -> mshop-store 业务服务
   -> mshop-db Mapper / MySQL
   -> Redis / 第三方淘客与支付平台
```

#### 2.6.2 后台调用链

```text
Vue 管理端
   -> mshop-admin
   -> mshop-shop / mshop-tool / mshop-log / mshop-gen
   -> mshop-store / mshop-system / mshop-db
   -> MySQL / Redis / Quartz / 外部平台
```

## 3. 模块级详细设计

## 3.1 `mshop-admin` —— 管理后台聚合入口模块

### 3.1.1 模块职责

- 提供后台认证、菜单、角色、用户、部门、岗位等系统管理能力
- 集成 Quartz 定时任务管理
- 集成在线用户、访问统计、Redis 监控、日志监控
- 聚合 `mshop-shop`、`mshop-tool`、`mshop-log`、`mshop-gen` 暴露的后台接口
- 作为管理端唯一运行入口之一

### 3.1.2 关键设计点

- 使用 Spring Security + JWT 进行后台认证
- `SecurityConfig` 会自动扫描 `@AnonymousAccess` 放行匿名接口
- `SwaggerConfig` 统一配置后台文档、鉴权头与 Knife4j 页面
- `TaskFullRunConfig` 从 `quartz.full` 读取“仅主节点执行”的任务列表

## 3.2 `mshop-app` —— 移动端 API 入口模块

### 3.2.1 模块职责

- 面向 APP / H5 / 小程序提供统一 API
- 用户、商品、订单、支付、营销、本地生活、第三方淘客等核心入口都集中在此模块
- 使用自定义拦截器进行用户鉴权与权限校验
- 通过 `DruidConfig` 接入主从数据源与动态切换

### 3.2.2 安全与访问控制

- `AppServer` 开启 `@EnableCaching`
- `InterceptorConfig` 注册：
  - `PermissionInterceptor`
  - `UserInterceptor`
- 基于注解控制：
  - `@AuthCheck`：需要登录
  - `@UserCheck`：需要用户上下文
- 资源映射：
  - `/avatar/**`
  - `/file/**`

## 3.4 `mshop-store` —— 商城通用业务域模块

### 3.4.1 模块职责

这是整个系统的核心业务域模块，承担：

- 用户、商品、订单、营销、支付、淘客、联盟订单等核心服务
- 与淘宝、京东、拼多多、抖音、唯品会、美团、饿了么等第三方平台适配
- 为 `mshop-app`、`mshop-shop`、`mshop-redis`、`mshop-mp` 提供统一业务能力

### 3.4.2 核心能力

| 领域 | 说明 |
|---|---|
| 商品域 | 商品查询、推荐、详情、评论、分享海报 |
| 订单域 | 下单、支付、状态流转、物流、售后 |
| 用户域 | 会员等级、分销、收益、实名、银行卡、提现 |
| 淘客域 | 大淘客、好单库、联盟转链、订单采集 |
| 支付域 | 多支付通道适配与路由 |
| 外卖/本地生活 | 美团、饿了么、抖音本地生活 |

### 3.4.3 关键实现点

- `DataokeConfig` 定义 `RestTemplate` 与 HTTP 超时配置（15 秒）
- `PddConfig`、`JdConfig`、`TbConfig` 等对接不同联盟平台
- 包含大量系统级 JAR 依赖：PDD SDK、淘宝 SDK、京东 SDK、银盛 SDK、饿了么 SDK、Meituan SDK
- 支付实现涉及：支付宝、微信、AdaPay、银盛、通联等

### 3.4.4 风险说明

当前 `mshop-store/pom.xml` 使用多个 `systemPath` 依赖，部署与 CI/CD 风险较高：

- 强依赖本地 `lib/` 目录
- 跨环境构建不稳定
- 不利于制品仓库统一管理

建议后续迁移至：

- 私有 Maven 仓库
- Nexus / Artifactory
- 或统一本地安装脚本

## 3.9 `mshop-redis` —— Redis 监听与初始化模块

### 3.9.1 模块职责

- 系统启动时将配置表同步到 Redis
- 监听 Redis Key 过期事件，触发订单与拼团相关业务动作

### 3.9.2 关键实现

#### A. 启动初始化

`RedisKeyInitialization` 在服务启动时执行：

- 从 `MwSystemConfig` 表读取配置
- 排除指定 JSON 类型配置项
- 将系统配置同步写入 Redis

#### B. 过期监听

`RedisKeyExpirationListener` 监听 `__keyevent@{db}__:expired`：

- 淘宝 PID 绑定过期后回收到 Redis 池
- 未支付订单超时自动取消
- 待收货订单超时自动收货
- 拼团超时自动取消与退款处理

### 3.9.3 关键配置

| 配置项 | 说明 |
|---|---|
| `spring.redis.database` | Redis DB 索引，用于监听 topic 拼接 |
| Redis `notify-keyspace-events` | 必须启用过期事件通知 |

## 6. 部署与运维建议

## 6.1 部署前必须确认

1. 外置 `application*.yml` 是否与环境一致
2. MySQL / Redis 地址、端口、账号、DB 是否正确
3. Redis 是否开启 Key 过期事件通知
4. `file.path`、`file.avatar`、合同目录是否存在且可写
5. 所有第三方密钥是否已替换为生产密钥
6. `quartz.main=true` 是否只在一台管理节点上开启
7. `tb.pid.pool`、`kuCid`、支付通道证书是否已配置完整

## 6.2 当前源码中需要重点治理的问题

### A. 配置安全

当前开发配置文件中存在数据库、Redis、第三方平台密钥等敏感信息示例，应改为：

- 环境变量
- 外置配置中心
- KMS / Secret Manager
- 加密配置

### B. 文档与配置不一致

- `RUNNING.md` 与 `application.yml` 的端口配置存在差异
- 应统一“文档端口、脚本端口、配置端口”

### C. 系统依赖管理风险

- 多个 `systemPath` JAR 不利于标准化构建
- 建议迁移到 Maven 私服

### D. 接口扫描范围较大

- 由于启动类都位于 `com.mailvor` 根包，依赖模块中的 Controller/Config 都可能被扫描
- 建议在服务设计层面明确“聚合型启动模块”的暴露边界

## 8. 文档结论

从当前源码结构看，系统采用了“**双启动入口 + 多业务模块聚合**”的设计：

- `mshop-admin` 负责后台、系统、监控、任务与业务后台管理
- `mshop-app` 负责移动端、商品、订单、支付、淘客与本地生活 API
- `mshop-store` 是真正的核心业务域
- `mshop-system` / `mshop-db` / `mshop-tool` / `mshop-log` / `mshop-redis` 则提供底层基础设施支撑

该设计的优点是：

- 业务分层较清晰
- 能快速复用公共域能力
- 易于支撑后台与移动端双场景

当前最需要优化的方向是：

- 配置安全
- 构建标准化
- 接口文档规范化
- 模块暴露边界治理
